# MVP 场景闭环 PRD

## 1. 目标

MVP 的目标不是一次性做完整企业级经营决策平台，而是验证一个核心闭环：

> 从数据和文档中生成面向大模型推理的场景本体，并让 Agent 在该场景下完成可追溯、可评测的经营或合规判断。

MVP 必须证明以下假设：

1. 业务文档和结构化数据 schema 可以被大模型转成可审核的本体草案。
2. 场景层可以有效缩小推理上下文。
3. 原子规则可以被大模型稳定组装成推理链。
4. Agent 可以调用工具获取事实，而不是依赖 prompt 中的静态信息。
5. 输出的推理轨迹足够透明，业务专家可以审核。
6. 评测集可以成为本体迭代的质量闸门。

## 2. MVP 推荐场景

建议第一期选择“运输合规审查”作为样板场景，同时将“锂电行业生产计划排程”作为第二个重点样板场景储备。

判断原则：

- 第一场景优先验证本体、文档规则、结构化数据、Agent 推理轨迹和评测闭环。
- 第二场景优先验证经营决策能力、复杂约束、跨系统数据调度、skills 调用和人机协同。
- 高风险行业场景必须先做“审查建议”和“辅助决策”，不直接自动执行最终业务动作。

### 2.1 第一推荐场景：运输合规审查

原因：

- 规则边界相对清晰；
- 对象数量有限；
- 有结构化数据：物料、BOM、工厂、订单；
- 有文档规则：危险品、原产国、运输条件；
- 容易构造评测集；
- 推理链包含多跳和条件叠加，足以验证平台核心主张；
- 风险高但可以先做建议，不直接自动执行。

样板问题：

> 成品 A 包含 B，B 包含 C，C 是危险品，A 原产国为中国。A 出库运输前是否需要特殊运输条件？

期望输出：

- 匹配场景：运输合规审查；
- 查询事实：BOM、危险品标志、原产国；
- 套用规则：危险性一跳传递、产地叠加触发；
- 推理轨迹：C → B → A → 申报要求；
- 结论：A 需要危险品出境申报；
- 证据：文档条款、数据查询来源；
- 建议动作：创建合规复核任务或提示物流专员。

### 2.2 第二推荐场景：锂电行业生产计划排程

结论：这个场景非常适合本平台，但不建议作为第一个极小 MVP。它比运输合规更能体现经营决策平台价值，但复杂度也明显更高。

适合的原因：

- 业务对象丰富：订单、电芯型号、BOM、工序、产线、设备、班次、人员、物料、库存、工装夹具、质检规则；
- 关系天然复杂：订单需求关联物料，物料关联 BOM，BOM 关联工序，工序关联产线和设备，设备受产能、稼动率、维护计划约束；
- 规则大量场景化：急单插单、换型损耗、产线兼容性、批次追溯、最小生产批量、交期优先级、库存安全水位；
- 强依赖结构化数据：ERP、MES、WMS、APS、设备台账、质量系统；
- 适合 skill 调用：产能测算、物料齐套检查、排程优化、瓶颈识别、交期模拟；
- 输出天然需要可解释：为什么这张订单延后，为什么某产线成为瓶颈，为什么建议调整生产顺序。

它尤其适合验证本平台的一个关键取舍：

> 大模型不直接求解复杂排程优化，而是负责识别场景、组装约束、调用排程或计算 skill、解释结果和生成可追溯的调整建议。

样板问题：

> 下周有 3 个高优先级电芯订单，当前 A 产线周三停机维护，正极材料库存不足 2 天，是否需要调整生产计划？如果需要，应该如何调整？

期望输出：

- 匹配场景：锂电生产计划排程；
- 查询事实：订单交期、BOM、物料库存、产线能力、设备维护计划、当前排程；
- 激活约束：交期优先、物料齐套、产线兼容、维护窗口、换型成本、最小批量；
- 调用 skills：物料齐套检查、产能测算、排程模拟、瓶颈识别；
- 推理轨迹：缺料影响哪些订单，维护影响哪些工序，哪些订单可提前或转线；
- 结论：给出推荐排程调整方案和风险；
- 证据：数据查询来源、规则来源、模拟结果；
- 建议动作：生成排程调整草案，提交计划员审核。

最小对象：

```yaml
objects:
  ProductionOrder:
    定义: 需要在指定交期前完成的生产需求。
    属性: [id, product_model, quantity, due_date, priority, status]
  ProductModel:
    定义: 锂电产品型号，例如某规格电芯或电池模组。
    属性: [id, name, chemistry_type, capacity_spec]
  Material:
    定义: 生产所需物料，例如正极材料、负极材料、电解液、隔膜、壳体。
    属性: [id, name, inventory_qty, safety_stock, lead_time]
  WorkCenter:
    定义: 可执行特定工序的产线、工段或设备组。
    属性: [id, name, compatible_products, capacity_per_shift, status]
  ProcessStep:
    定义: 生产工艺步骤，例如制浆、涂布、辊压、分切、卷绕、注液、化成、分容。
    属性: [id, name, sequence, required_work_center_type]
  MaintenanceWindow:
    定义: 设备或产线不可用的计划维护时间。
    属性: [work_center_id, start_time, end_time, reason]
```

最小关系：

```yaml
relations:
  requiresMaterial:
    定义: 某产品型号生产时需要某物料。
    方向: ProductModel -> Material
    场景条件: 在物料齐套检查和排程模拟场景下成立。
  followsProcess:
    定义: 某产品型号需要经过某工艺步骤。
    方向: ProductModel -> ProcessStep
    场景条件: 在生产计划排程场景下成立。
  canRunOn:
    定义: 某产品型号或工序可在某产线或工作中心执行。
    方向: ProcessStep -> WorkCenter
    场景条件: 仅当工作中心状态可用且满足产品兼容性时成立。
  blockedByMaintenance:
    定义: 某工作中心在维护窗口内不可排产。
    方向: WorkCenter -> MaintenanceWindow
    场景条件: 在维护窗口覆盖的时间段内成立。
```

最小规则：

```yaml
rules:
  R1_物料齐套约束:
    条件: 订单所需关键物料的可用库存低于订单需求量。
    结果: 标记该订单存在缺料风险，不应安排在物料到齐前生产。
  R2_维护窗口约束:
    条件: 某工序计划时间与工作中心维护窗口重叠。
    结果: 该排程不可行，需要调整时间或更换可兼容工作中心。
  R3_交期优先规则:
    条件: 多个订单竞争同一瓶颈工作中心，且其中一个订单优先级更高或交期更早。
    结果: 优先保障高优先级或更早交期订单。
  R4_换型成本提示:
    条件: 相邻生产任务产品型号差异导致换型时间超过阈值。
    结果: 标记排程存在换型损耗，建议合并相同或相近型号批次。
```

最小 skills：

```yaml
skills:
  query_production_orders:
    说明: 查询指定时间窗口内的生产订单。
  check_material_availability:
    说明: 根据订单、BOM 和库存检查物料是否齐套。
  query_work_center_capacity:
    说明: 查询产线或设备组产能、状态和维护窗口。
  simulate_schedule:
    说明: 在给定约束下生成一个候选排程或评估排程可行性。
  identify_bottlenecks:
    说明: 根据订单、工序和产能识别瓶颈工作中心。
```

MVP 边界：

- 第一版只做“排程建议”和“可行性审查”，不直接写回 MES 或 APS。
- 大模型不承担精确排程优化，必须通过 `simulate_schedule` 或类似 skill 调用外部计算。
- 输出必须说明使用了哪些约束、哪些数据、哪些假设和哪些不确定性。
- 若存在缺料、维护冲突或产能不足，必须明确指出影响订单和建议处理方式。

评测重点：

- 场景匹配是否正确；
- 约束是否激活完整；
- 缺料、维护、产能冲突是否识别正确；
- skill 调用计划是否合理；
- 推荐调整是否符合业务规则；
- 推理轨迹是否能被计划员复核。

### 2.3 候选高价值场景：金融合规与风控审查

结论：金融合规和风控审查也非常适合，但更适合作为“高治理要求场景”，不建议在没有明确法规、制度、审计和责任边界前作为第一个 MVP。

适合的子场景：

- 客户准入尽调审查；
- 交易异常初筛；
- 授信材料完整性审查；
- 贷后风险预警解释；
- 反洗钱可疑线索辅助分析；
- 投顾或销售适当性审查；
- 内控制度匹配与例外检查。

适合的原因：

- 文档规则多：监管规定、内部制度、审批手册、风险政策；
- 证据要求强：每个结论都需要来源、事实和解释；
- 场景边界清楚：准入、交易、授信、贷后、适当性等可拆成独立场景；
- 非结构化材料多：合同、流水、尽调报告、公告、审计材料；
- 评测价值高：可用历史案例、拒绝原因、审批意见构造 benchmark。

主要风险：

- 具体监管规则会变化，需要持续更新和版本治理；
- 输出不能被误认为法律或监管最终结论；
- 高风险结论必须人工复核；
- 对证据、权限、脱敏和审计要求更高；
- 模型幻觉和遗漏事实的代价更大。

MVP 边界建议：

- 只做“辅助审查”和“风险提示”，不自动通过或拒绝客户、交易、授信。
- 所有结论必须引用制度条款、数据事实和推理轨迹。
- 先选一个窄场景，例如“授信材料完整性审查”或“客户准入初筛”，避免一开始覆盖全量风控。
- 若涉及具体监管条款，必须以企业提供的最新制度和法规文本为准，并进入知识库版本管理。

与锂电排程的对比：

| 维度 | 锂电生产计划排程 | 金融合规/风控审查 |
| --- | --- | --- |
| 业务价值 | 高，直接连接交期、产能、库存和成本 | 高，直接连接风险、合规和审批效率 |
| 数据依赖 | 强依赖 ERP/MES/WMS/APS 结构化数据 | 同时依赖结构化数据和大量文档材料 |
| 推理类型 | 多约束经营决策 + 工具计算 | 规则审查 + 证据链 + 风险判断 |
| 对本平台验证 | 很适合验证本体、skills、数据源和决策建议 | 很适合验证文档规则、审计、可追溯和权限 |
| 第一版难度 | 中高 | 高 |
| 自动化边界 | 建议生成排程草案，人工审核 | 只做辅助审查，人工最终判断 |
| 推荐阶段 | 第二个样板场景 | 有行业客户和制度材料后进入专项样板 |

优先级建议：

1. 第一场景：运输合规审查，用来打通本体、规则、数据、Agent、评测的最小闭环。
2. 第二场景：锂电生产计划排程，用来证明平台能处理真实经营决策和复杂约束。
3. 第三场景：金融合规或风控审查，用来验证高治理、高审计、高证据要求场景。

## 3. 用户角色

### 3.1 业务专家

负责上传政策文档、审核本体草案、确认规则和评测用例。在锂电排程场景中，该角色通常是计划经理、工艺专家或供应链负责人；在金融场景中，该角色通常是合规、风控或内控专家。

### 3.2 数据负责人

负责接入物料、BOM、工厂和订单数据，确认字段含义与数据质量。

### 3.3 经营分析或合规人员

使用 Agent 提问，查看结论、证据和推理过程。

### 3.4 平台管理员

配置模型、连接器、权限、评测运行和发布流程。

## 4. 范围

### 4.1 In Scope

MVP 包含：

- 文档上传；
- 文档解析和切片；
- 数据源 schema 接入；
- 本体草案生成；
- 本体人工审核；
- 场景定义；
- 原子规则定义；
- 数据查询 skill；
- Agent 场景推理；
- 结构化推理轨迹；
- 评测用例管理；
- 回归测试；
- 简单决策工作台。

### 4.2 Out of Scope

MVP 暂不包含：

- 多租户商业化计费；
- 完整企业权限体系；
- 自动执行高风险动作；
- 多场景复杂编排；
- 实时流式事件触发；
- 大规模知识图谱可视化；
- 复杂指标平台；
- 多模型路由优化；
- 插件市场。

## 5. 关键用户流程

### 5.1 初始化场景

1. 管理员创建工作空间。
2. 管理员创建“运输合规审查”场景。
3. 数据负责人接入物料主数据和 BOM 数据源。
4. 业务专家上传运输合规政策文档。
5. 系统生成候选对象、关系、规则和场景激活配置。
6. 业务专家审核并发布本体版本。
7. 业务专家录入或生成评测用例。
8. 系统运行评测，通过后允许该场景进入可用状态。

### 5.2 用户提问与推理

1. 用户输入：物料 A 是否需要危险品出境申报？
2. Agent 匹配场景：运输合规审查。
3. Agent 加载该场景激活的对象、关系、规则和 tools。
4. Agent 调用 `query_material_bom` 查询 A 的 BOM。
5. Agent 递归查询直接子件，直到达到配置的最大深度或无子件。
6. Agent 调用 `query_material_master` 查询危险品标志和原产国。
7. Agent 按原子规则生成推理链。
8. Agent 输出结论、轨迹、证据和建议动作。
9. 用户反馈结论是否正确。
10. 系统可将反馈转为待审核评测用例。

### 5.3 本体变更与回归

1. 业务专家修改规则或场景。
2. 系统创建新的本体草案版本。
3. 系统识别受影响评测集。
4. 系统运行回归测试。
5. 若通过，允许发布。
6. 若失败，展示失败用例和推理轨迹差异。

## 6. 功能需求

### 6.1 文档上传与切片

用户可以上传 PDF、Word、Markdown 或文本文件。

系统需要：

- 抽取文本；
- 按语义切片；
- 识别规则、定义、例外条件；
- 保留来源位置；
- 支持人工修正切片类型；
- 将切片链接到本体元素。

验收标准：

- 每个规则候选都能追溯到文档片段；
- 文档片段可在推理结果中引用；
- 用户可以标记某个切片为“规则来源”或“定义来源”。

### 6.2 数据源接入

MVP 可先支持 CSV 或本地数据库表，后续再扩展完整 MCP 连接器。

最小数据表：

```yaml
material_master:
  material_id: string
  material_name: string
  origin_country: string
  is_hazardous: boolean

bom:
  parent_material_id: string
  child_material_id: string
  quantity: number
  bom_version: string
  valid_from: date
  valid_to: date | null
```

需要暴露两个 skill：

```yaml
query_material_master(material_id) -> material_master
query_material_bom(material_id, bom_version?) -> child_materials
```

验收标准：

- Agent 可以通过 skill 查询物料事实；
- 查询结果进入推理轨迹；
- 数据来源在最终答案中可见。

### 6.3 本体生成

系统根据文档和 schema 自动生成候选本体元素。

候选元素包括：

- ObjectType：Material、Factory、ShipmentOrder；
- Property：origin_country、is_hazardous；
- RelationType：hasPart、producedIn；
- Rule：危险性一跳传递、产地叠加触发；
- Scenario：运输合规审查；
- Skill：查询 BOM、查询物料主数据。

验收标准：

- 候选本体以结构化形式展示；
- 每个候选元素带自然语言定义；
- 每个候选元素带来源；
- 用户可接受、编辑、拒绝候选元素。

### 6.4 本体审核与发布

业务专家可以审核本体草案。

状态流：

```text
draft -> pending_review -> approved -> production
draft -> rejected
production -> deprecated
```

验收标准：

- 未审核元素不能进入生产版本；
- 发布本体版本前必须运行评测；
- 发布后 Agent 绑定到具体版本。

### 6.5 场景配置

场景配置包括：

- 场景定义；
- 触发问题模式；
- 激活对象；
- 激活关系；
- 激活规则；
- 允许调用的 skills；
- 输出结构要求；
- 最大推理深度；
- 是否允许自动动作。

运输合规审查默认配置：

```yaml
max_bom_depth: 5
require_evidence: true
require_reasoning_trace: true
allow_external_action: false
```

验收标准：

- Agent 只加载当前场景激活元素；
- 场景匹配结果显示给用户；
- 用户可查看场景激活了哪些规则和工具。

### 6.6 Agent 推理

Agent 必须输出结构化结果：

```yaml
matched_scenario: string
facts_used:
  - fact: string
    source: string
rules_applied:
  - rule_id: string
    reason: string
reasoning_trace:
  - step: number
    description: string
    inputs: list
    output: string
conclusion:
  text: string
  confidence: number
evidence:
  - source: string
    excerpt: string
uncertainty:
  - string
recommended_actions:
  - string
```

验收标准：

- 结论不能脱离推理轨迹单独出现；
- 每个规则应用步骤必须引用规则 ID；
- 数据事实必须引用查询来源；
- 若缺少事实，必须明确说明缺失项；
- 若超出最大 BOM 深度，必须说明截断风险。

### 6.7 评测集

MVP 必须支持手工创建和从反馈生成评测用例。

评测维度：

- 场景是否匹配；
- 规则是否选对；
- 中间推理是否完整；
- 最终结论是否正确；
- 证据是否正确；
- 输出结构是否合规。

验收标准：

- 可运行单条评测；
- 可运行整个场景评测集；
- 展示通过率和失败原因；
- 本体变更后可对比两次评测结果。

## 7. 页面需求

### 7.1 场景工作台

显示：

- 场景名称和定义；
- 激活对象；
- 激活关系；
- 激活规则；
- 绑定 skills；
- 评测通过率；
- 最近 Agent 运行记录。

### 7.2 本体编辑器

支持查看和编辑：

- 对象；
- 属性；
- 关系；
- 规则；
- 场景；
- 来源证据；
- 审核状态。

MVP 可先用表单和 YAML/JSON 双视图，不需要复杂图谱可视化。

### 7.3 文档知识库

显示：

- 文档列表；
- 切片列表；
- 切片类型；
- 链接的本体元素；
- 审核状态。

### 7.4 Agent 决策台

包含：

- 问题输入；
- 场景匹配结果；
- 查询到的事实；
- 推理轨迹；
- 结论；
- 证据；
- 反馈按钮；
- 保存为评测用例。

### 7.5 评测页面

显示：

- 评测集列表；
- 用例列表；
- 单用例结果；
- 总体通过率；
- 失败原因；
- 两次运行对比。

## 8. 非功能需求

### 8.1 可追溯性

每次 Agent 输出必须保存：

- 用户问题；
- 匹配场景；
- 本体版本；
- 模型版本；
- prompt 版本；
- 工具调用；
- 数据返回；
- 规则应用；
- 证据引用；
- 最终结论。

### 8.2 可复现性

评测运行需要记录模型、参数、本体版本和输入数据快照。否则结果不可比较。

### 8.3 安全性

MVP 至少需要：

- 工作空间级用户隔离；
- 数据源只读；
- 高风险动作禁用；
- Agent 输出不自动写回外部系统；
- 敏感字段可标记。

### 8.4 性能

MVP 目标：

- 单次普通问答在 30 秒内完成；
- BOM 深度不超过 5；
- 单场景激活规则不超过 50 条；
- 单次上下文本体切片不超过模型可承受窗口的 40%。

## 9. 成功指标

### 9.1 产品指标

- 业务专家完成一个场景配置的时间；
- 候选本体审核通过率；
- 单个问题平均分析耗时；
- 用户反馈正确率；
- 保存为评测用例的比例。

### 9.2 推理质量指标

- 场景匹配准确率 >= 95%；
- 规则选择准确率 >= 90%；
- 最终结论准确率 >= 90%；
- 推理轨迹完整率 >= 85%；
- 结构化输出合规率 >= 98%；
- 同一用例重复运行结论一致率 >= 90%。

这些数字是 MVP 假设值，需要通过实验校准。

## 10. 第一批评测用例设计

建议至少准备 30 条用例：

- 10 条简单一跳危险品判断；
- 8 条多跳 BOM 传递；
- 5 条原产国叠加触发；
- 3 条非中国原产国不触发；
- 2 条缺失危险品标志；
- 2 条 BOM 超过最大深度或存在环。

示例：

```yaml
id: evalcase.transport.001
question: A 包含 B，B 包含 C，C 是危险品，A 原产国中国。A 是否需要危险品出境申报？
facts:
  - A hasPart B
  - B hasPart C
  - C.is_hazardous = true
  - A.origin_country = 中国
expected:
  scenario: 运输合规审查
  conclusion: A 需要危险品出境申报
  trace_must_include:
    - C 危险 -> B 含危险子件
    - B 含危险子件 -> A 含危险子件
    - A 原产国中国 -> 触发危险品出境申报
```

## 11. 技术实现建议

### 11.1 第一版存储

MVP 可以使用关系型数据库保存本体和评测数据，使用 JSONB 保存结构化规则和输出契约。

推荐：

- PostgreSQL：核心数据；
- pgvector 或独立向量库：文档切片检索；
- 对象存储：原始文档；
- 本地或轻量任务队列：评测运行。

### 11.2 第一版 Agent 编排

建议先用确定的运行管线，而不是一开始做完全自主 Agent：

```text
输入问题
-> 场景分类
-> 加载场景本体切片
-> 生成取数计划
-> 调用工具
-> 规则应用推理
-> 结构化输出
-> 保存轨迹
```

这样更容易评测和调试。

### 11.3 第一版规则格式

规则同时保存自然语言和结构化字段。

自然语言给大模型理解，结构化字段用于展示、检索、评测和部分校验。

## 12. 里程碑

### M1：基础数据与文档接入

- 上传文档；
- 导入 CSV 或连接数据库；
- 查看 schema；
- 创建场景。

### M2：本体生成与审核

- 自动生成对象、关系、规则；
- 人工审核；
- 发布本体版本。

### M3：Agent 推理闭环

- 用户提问；
- 场景匹配；
- 工具取数；
- 规则组装；
- 结构化推理轨迹。

### M4：评测闭环

- 创建评测用例；
- 运行评测；
- 展示失败；
- 本体变更回归。

### M5：样板演示

- 使用一组真实或模拟数据；
- 完成运输合规审查端到端演示；
- 输出平台下一阶段需求。

## 13. 待决策问题

1. MVP 是否必须接真实 MCP，还是先用本地 CSV/DB 模拟 MCP 能力？
2. 第一版是否只支持运输合规审查，还是同时准备锂电生产计划排程作为第二样板场景？
3. 规则推理是否允许调用外部计算函数，还是第一版保持纯 LLM 组装？
4. 评测打分先用规则匹配，还是引入 LLM-as-judge？
5. 本体编辑器第一版采用表单优先，还是 YAML 优先？
6. 输出是否需要支持报告导出？
7. 是否需要从第一版开始做权限隔离？

## 14. MVP 不做什么

为了保持闭环清晰，第一版不要做以下内容：

- 不做通用低代码平台；
- 不做复杂图谱编辑器；
- 不做所有企业系统连接器；
- 不做自动执行真实业务动作；
- 不做跨行业通用规则市场；
- 不做“无限自主”的 Agent；
- 不做没有评测约束的聊天式问答。

MVP 的唯一重点是证明：面向大模型推理的场景本体，是否真的能让企业经营判断变得更可控、更透明、更容易迭代。
