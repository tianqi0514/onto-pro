# 金融客户场景与评测方案

## 1. 文档目标

本文面向江苏资产/融保贷客户的第一批落地场景，回答三个问题：

1. 这个客户最重要的业务场景是什么。
2. Onto Pro 应如何通过场景本体、原子规则、skills、知识库和结构化推理轨迹实现这些场景。
3. 每个场景如何设计 P0/P1/P2 评测，用来验证平台能力边界，尤其是本体抽取、关系推理、事件理解和多跳推理在第几层开始失真。

本文不是普通业务系统 PRD。客户已有流程系统、报表系统、合同模板和材料清单；Onto Pro 的关键价值是把这些材料转成大模型可消费的业务语义层，让 AI 能解释“为什么”、能引用证据、能沉淀规则，并能被持续评测。

## 3. 客户理解

客户核心业务包括对公不良资产处置、个贷不良资产处置、融资租赁、保理等。现场纪要中明确提到，客户正在重建覆盖流程、数据、报表、档案的统一业务系统；AI 不应替代这个业务系统，而应在其旁边形成“AI 金融智能体/项目级 AI 工作台”，重点处理资料解析、报告生成、审查、上会问答、执行跟踪和经验沉淀。

客户当前已有一些流程知识图谱，但主要描述 `S01 立项 -> S02 尽调 -> S03 风险审查 -> ...` 这样的流程顺序。对大模型推理而言，这还不够。真正需要的是：

- 业务对象层：项目、客户、合同、发票、应收账款、租赁物、抵押物、授信、还款计划、资产包。
- 状态层：立项中、尽调中、已审批、已放款、还款中、逾期、结清、核销。
- 规则层：风控规则、审批规则、额度规则、放款前置规则、合同条款规则。
- 事件层：舆情事件、发票作废、回款异常、逾期、额度占满、担保失效、法院执行、查封冻结。
- 决策层：风险评分、授信建议、资产分类、处置策略、上会结论、是否放款。

因此，第一个客户的落地重点不应是“做一个流程图谱”，而应是做一个以场景为中心的金融业务语义操作层。

## 4. 总体实现框架

### 4.1 核心分工

- 本体层：定义业务对象、状态、关系、规则、事件、指标、材料、合同条款和场景。
- Agent 层：匹配场景，选择本体切片，规划工具调用，组装原子规则，输出可追溯结论。
- Skill 层：执行 OCR、字段抽取、财务计算、额度计算、合同比对、材料完整性检查、报表生成、外部 API 查询。
- 评测层：验证场景匹配、对象抽取、关系抽取、规则选择、工具调用、推理轨迹和结论。
- 治理层：管理本体版本、Skill 版本、证据来源、人工反馈、回归测试和权限审计。

### 4.2 本体解析层级

为了专门测试“解析到第几层会失真”，建议统一采用以下层级：

| 层级 | 名称 | 说明 | 典型失真点 |
| --- | --- | --- | --- |
| L0 | 文档层 | 识别文件类型、来源、项目、阶段、页码、上传位置 | 扫描件/OCR 失败，文件名误导 |
| L1 | 实体字段层 | 抽取项目、主体、金额、日期、合同编号、票据号码、抵押物坐落等 | 金额单位、日期、主体简称混淆 |
| L2 | 关系层 | 建立项目-主体、项目-合同、合同-抵押物、应收账款-买卖方等关系 | 多角色主体混淆，主从合同关系错连 |
| L3 | 状态层 | 判断项目、合同、应收账款、登记、面签、还款的状态 | “已签署/已生效/已盖章/已登记”状态混用 |
| L4 | 规则层 | 把制度、控制点、合同条款转成原子规则并正确激活 | 规则条件漏项，软提示/硬拦截混淆 |
| L5 | 跨文档一致性层 | 校验合同、审批表、转让通知、评估报告、票据、台账之间的一致性 | 同名主体、不同金额、不同版本无法消歧 |
| L6 | 事件层 | 识别舆情、诉讼、查封、逾期、担保失效、回款异常等事件及影响 | 事件与项目/主体关联错误 |
| L7 | 决策层 | 形成风险判断、放款建议、上会摘要、资产分类、处置策略 | 证据不足仍给确定结论，过度推断 |

第一阶段的产品成功不要求 L0-L7 全部稳定。更现实的目标是：L0-L4 在租赁/保理样板场景中稳定，L5 可在关键字段上通过，L6-L7 用少量高价值场景探索边界。

### 4.3 P0/P1/P2 测试定义

- P0：最小可用验证。输入结构清晰、文档可抽取、规则明确，验证平台能否完成场景匹配、核心实体抽取、单跳/两跳关系、原子规则应用和证据引用。
- P1：真实业务验证。输入来自多个文件和系统，包含跨文档一致性、多阶段状态、工具调用、规则组合，验证平台是否能稳定完成 3-5 跳推理。
- P2：能力边界验证。输入包含扫描件、缺失字段、冲突证据、隐性关联、事件变化、长链推理或开放式决策，观察平台在 L5-L7 哪一层开始失真。

## 6. 场景详情

### S1 项目材料归档与本体草案生成

目标：把一个融资租赁或保理项目的材料包自动归档到项目、阶段和材料类型，并生成可审核的本体草案。

客户痛点：材料分散在审批表、合同、征信报告、评估报告、票据、身份证明、营业执照、章程、面签表、抵押登记等文件中。人工归档和核对成本高，且扫描件/盖章件很多。

本体切片：

- 对象：Project、Document、BusinessMaterial、Party、Contract、Collateral、LeasedAsset、Receivable。
- 关系：Project hasMaterial Document；Document belongsToStage Stage；Document evidences Entity；Document generatedBy Party。
- 状态：required、uploaded、missing、recognized、needs_review、approved。
- 规则：必填材料缺失规则、材料阶段归属规则、同主体材料复用规则、扫描件需人工复核规则。

需要的 skills：

- `classify_document_type`
- `extract_document_metadata`
- `extract_project_id`
- `ocr_scanned_pdf_or_image`
- `check_material_completeness`
- `link_document_to_ontology_element`

输出结构：

- 项目材料清单完成率。
- 缺失材料列表。
- 文件到本体元素的映射表。
- 抽取置信度和需人工复核项。
- 每个材料对应的来源文件和页码。

测试设计：

| 等级 | 测试内容 | 通过标准 | 观察边界 |
| --- | --- | --- | --- |
| P0 | 用 `租赁-材料清单.xlsx` 和清水绿岸项目中可抽取的审批表/评估报告，识别立项、审批、合同、实施、放款、贷后材料 | 文档类型识别准确率 >= 90%，必填材料缺失识别准确 | L0-L1 是否稳定 |
| P1 | 混合 PDF、DOCX、XLSX、文件夹路径，自动把同一项目材料关联到 `2025-RZ053/2025-RZ048` 项目 | 项目归属准确率 >= 85%，同名/编号不一致能提示冲突 | L2-L3 是否失真 |
| P2 | 加入扫描合同、面签表、章程、身份证明、图片文件和模糊文件名 | 能把低置信度材料标为 needs_review，不凭空抽取 | OCR 失败时是否产生幻觉 |

能力边界假设：纯文本和原生 PDF 可稳定到 L2；扫描件若无 OCR/版面模型，L1 就会失真；材料阶段归属可通过文件路径+材料清单稳定到 L3。

### S2 客户准入、工商舆情与黑名单核验

目标：对承租人、保理卖方、保理买方、担保人、抵押人等主体做准入核验，输出风险摘要和准入建议。

客户痛点：需要自动抓取工商、司法、涉诉、失信、限高、行政处罚、股权变更、经营异常、被执行人和舆情信息；同时需要能解释命中了哪些风险。

本体切片：

- 对象：Party、CreditReport、ExternalRiskSignal、BlacklistRecord、PublicOpinionEvent、LegalCase。
- 关系：Party hasCreditReport；Party involvedIn LegalCase；Party hasRiskSignal；Party controls Party。
- 状态：normal、watch、blocked、manual_review。
- 规则：黑名单命中硬拦截、重大负面舆情软预警、失信被执行硬拦截、经营异常软预警。
- 事件：工商变更、司法执行、负面舆情、行政处罚。

需要的 skills：

- `query_business_registry`
- `query_blacklist`
- `query_credit_report`
- `query_legal_cases`
- `query_public_opinion`
- `summarize_party_risk`

输出结构：

- 主体准入结论：通过/预警/拒绝/需人工复核。
- 风险信号列表。
- 证据来源和时间。
- 触发规则和规则等级。
- 对项目的影响。

测试设计：

| 等级 | 测试内容 | 通过标准 | 观察边界 |
| --- | --- | --- | --- |
| P0 | 对一个主体征信报告或模拟工商记录抽取名称、统一社会信用代码、查询时间、信用状态 | L1 字段准确率 >= 95%，能引用原始报告 | 字段抽取是否稳定 |
| P1 | 对项目多主体同时核验，区分承租人、担保人、抵押人不同角色 | 角色不混淆，风险结论能按角色聚合 | L2 角色关系是否稳定 |
| P2 | 加入同名公司、历史名称、股权变更、负面舆情和外部 API 冲突 | 能提示冲突和不确定性，不直接给确定结论 | L6 事件和冲突处理边界 |

能力边界假设：主体字段抽取可稳定到 L1；多角色关系到 L2 需要项目上下文；舆情事件影响评估到 L6 需要时间、来源可信度和人工复核机制。

### S3 融资租赁尽调与风险审查

目标：围绕融资租赁售后回租项目，自动生成尽调/风控审查辅助结论，覆盖主体资信、租赁物、抵押物、财务指标、增信措施和需关注事项。

样板材料：清水绿岸项目材料、港城建设风险审查意见。

本体切片：

- 对象：LeasingProject、BusinessPlan、Lessee、Guarantor、Mortgagor、LeasedAsset、Collateral、FinancialStatement、CreditReport、RiskReviewOpinion。
- 关系：Project hasLessee；Project hasGuarantor；Project hasLeasedAsset；Project hasCollateral；Collateral ownedBy Mortgagor；GuaranteeContract guarantees LeasingContract。
- 规则：资产负债率预警、租赁物评估覆盖、抵押物孰低取值、第二顺位抵押风险、区域存量投放预警、实缴资本比例预警。
- 指标：资产负债率、流动比率、速动比率、评估覆盖率、抵押率、区域集中度、融资期限、综合收益率。

需要的 skills：

- `extract_project_basic_info`
- `extract_financial_statement_metrics`
- `calculate_debt_asset_ratio`
- `calculate_collateral_coverage`
- `compare_external_internal_valuation`
- `generate_risk_review_summary`

输出结构：

- 项目基础信息摘要。
- 主体资信结论。
- 租赁物和抵押物风险。
- 财务指标和阈值触发。
- 需关注事项。
- 风险审查建议。
- 推理轨迹和证据清单。

测试设计：

| 等级 | 测试内容 | 通过标准 | 观察边界 |
| --- | --- | --- | --- |
| P0 | 从项目审批表抽取融资金额、期限、收益率、承租人、租赁物、资产负债率 | 关键字段准确率 >= 95% | L1 字段抽取 |
| P1 | 用审批表+风险审查意见+评估报告，判断抵押物净值是否覆盖融资金额、第二顺位是否触发风险提示 | 规则选择准确，计算结果可复核 | L4-L5 规则组合 |
| P2 | 加入扫描财报、多个估值口径、项目编号不一致、清水绿岸 `RZ053/RZ048` 命名冲突 | 能识别冲突并要求人工确认 | L5 冲突证据边界 |

能力边界假设：融资租赁项目可作为第一批高价值 MVP，因为对象和规则已经比较清晰；真正难点在财报扫描件、估值口径、多版本项目编号和“需关注事项”的开放式归纳。

### S4 保理贸易背景真实性与四符合审查

目标：校验保理业务中应收账款是否真实、合法、可转让，是否符合合同、发票、物流/交付、验收/确权四类证据闭环。

样板材料：古城建设 2024 年 1 号保理项目知识图谱、保理合同模板、应收账款转让通知书、保理合规控制点。

本体切片：

- 对象：FactoringProject、Seller、Buyer、FactoringCompany、Receivable、Invoice、BaseTradeContract、DeliveryRecord、AcceptanceRecord、ReceivableTransferNotice、ReceivableTransferConfirmation。
- 关系：Seller holds Receivable；Buyer owes Receivable；Receivable generatedBy BaseTradeContract；Invoice supports Receivable；Receivable transferredTo FactoringCompany。
- 状态：generated、confirmed、transferred、financed、repaid、overdue、disputed。
- 规则：应收账款不得重复融资、账期需覆盖融资期限、发票/合同/验收金额一致、债权转让通知需送达并回执、商业纠纷触发反转让风险。

需要的 skills：

- `extract_receivable_facts`
- `verify_invoice`
- `check_trade_evidence_consistency`
- `query_cnab_registration`
- `detect_duplicate_financing`
- `calculate_factoring_ltv`

输出结构：

- 应收账款真实性结论。
- 四符合检查表。
- 账期/金额/融资比例检查。
- 重复融资和权利瑕疵风险。
- 放款前需补充材料。

测试设计：

| 等级 | 测试内容 | 通过标准 | 观察边界 |
| --- | --- | --- | --- |
| P0 | 从保理项目摘要中抽取卖方、买方、担保方、应收账款金额、融资金额、到期日 | 字段和角色准确率 >= 90% | L1-L2 |
| P1 | 对合同、发票、转让通知、回执、中登登记做四符合和账期匹配 | 能输出逐项通过/不通过和证据 | L4-L5 |
| P2 | 制造发票金额与合同金额不一致、买方回执缺失、应收账款已质押或重复融资 | 能发现冲突并阻断/预警 | L5 真实性校验边界 |

能力边界假设：保理场景比租赁更能验证跨文档一致性；P2 很适合暴露大模型在“多凭证闭环”中的漏项问题。

### S5 合同模板审查与跨合同一致性校验

目标：对融资租赁合同、保理合同、保证合同、抵押合同、转让协议、所有权转移确认书、应收账款转让通知书等进行模板比对、条款完整性审查和跨合同一致性校验。

- 关系：GuaranteeContract guarantees PrincipalContract；MortgageContract secures PrincipalContract；TransferNotice references Receivable；Contract hasClause Clause。
- 状态：draft、pending_signature、signed、effective、changed、abnormal。
- 规则：主从合同编号一致、主体名称一致、融资金额一致、担保范围覆盖主债权、回款账户一致、管辖法院条款存在、加速到期条款存在、保证期间满足要求。

需要的 skills：

- `compare_contract_with_template`
- `extract_contract_clauses`
- `extract_contract_parties`
- `check_cross_contract_consistency`
- `detect_clause_deletion_or_blank`
- `generate_contract_review_report`

输出结构：

- 模板差异列表。
- 高/中/低风险条款。
- 跨合同字段一致性报告。
- 缺失条款和留白字段。
- 证据定位到合同页码/条款。

测试设计：

| 等级 | 测试内容 | 通过标准 | 观察边界 |
| --- | --- | --- | --- |
| P0 | 从模板合同中抽取合同类型、主体、金额、日期、附件清单、关键条款 | L1 字段和条款识别准确 | L1-L2 |
| P1 | 比对主合同、保证合同、抵押合同的主体、主合同编号、担保范围、抵押物清单 | 能列出一致/不一致字段 | L5 |
| P2 | 加入盖章扫描件、手写修改、空白占位符、附件缺失、条款被删改 | 能标出低置信度并要求人工复核 | OCR+版面+条款理解边界 |

能力边界假设：电子模板可稳定；盖章扫描件若无 OCR 和版面对齐能力，会在 L1-L2 失真；跨合同一致性必须由结构化抽取和规则校验共同完成，不能只靠自由问答。

- `extract_attention_items`

## 7. 第一批 MVP 推荐

建议第一批不要同时做 14 个场景。应选择一个能跑通闭环、又能展示客户价值的组合：

1. S1 项目材料归档与本体草案生成。
2. S3 融资租赁尽调与风险审查。
3. S5 合同模板审查与跨合同一致性校验。
4. S6 放款前置条件与实施管控。
5. S7 上会材料摘要与领导问答。

选择原因：

- 客户已有清水绿岸和港城建设租赁项目材料，样板数据充足。
- 租赁对象、关系、材料、规则已经比较清晰。
- 能覆盖 L0-L5 的核心能力边界。
- 能快速证明“不是流程图谱，而是业务语义 + 推理轨迹 + 证据”的差异。
- 保理场景可作为并行第二样板，用古城建设项目验证 S4 和 S8。

## 9. 能力边界指标

建议所有场景统一观测以下指标：

| 指标 | 含义 |
| --- | --- |
| 场景匹配准确率 | 用户问题或任务是否匹配到正确 Scenario |
| 文档类型识别准确率 | L0 是否能识别材料类型和阶段 |
| 实体抽取准确率 | L1 字段是否正确 |
| 关系抽取准确率 | L2 关系是否正确 |
| 状态判断准确率 | L3 状态是否正确 |
| 规则选择准确率 | L4 是否激活正确规则 |
| 跨文档一致性检出率 | L5 是否发现冲突和缺失 |
| 事件关联准确率 | L6 是否把事件关联到正确主体/项目 |
| 决策结论准确率 | L7 建议是否符合专家预期 |
| 证据引用准确率 | 是否引用到正确文件、页码、条款或字段 |
| 不确定性表达率 | 证据不足时是否明确说明 |
| 幻觉率 | 无证据事实占比 |
| 多轮一致性 | 连续追问中结论和证据是否漂移 |

## 10. 预期失真边界

基于当前材料形态，初步判断：

- L0-L1：DOCX、XLSX、原生 PDF 可较稳定；扫描 PDF、图片、公章、人脸核验会明显依赖 OCR/视觉模型。
- L2：显式业务关系可稳定，例如项目-承租人、项目-担保人、合同-主合同；隐性关联关系需要外部数据和图检索。
- L3：流程状态和材料状态可稳定；合同状态容易因“草稿/已签署/已盖章/已生效/已登记”混用而失真。
- L4：阈值规则和门控规则可稳定；自然语言“需关注事项”转规则需要人工审核。
- L5：跨文档一致性是第一批最重要的边界测试，尤其是金额、日期、主体、合同编号、账户、抵押物坐落和面积。
- L6：事件驱动风控需要外部 API、时间线和事件可信度；不能只靠静态材料。
- L7：上会建议、收购建议、五级分类建议必须以“辅助决策”输出，不能自动替代人工结论。

## 11. 与现有客户系统的关系

客户统一业务系统负责：

- 流程审批。
- 项目台账。
- 合同流转。
- 材料归档。
- 报表填报。
- 权限和审计。

Onto Pro 负责：

- 从文档和数据生成可审核的业务本体。
- 让 Agent 基于场景加载本体切片、规则、材料、证据和 skills。
- 对项目进行可解释的尽调、风控、合规、合同和上会辅助。
- 把专家反馈沉淀为规则、Skill 和评测用例。
- 持续回归验证本体变更是否破坏推理质量。

这个“双轨并行”对客户最现实：不改造核心业务系统，先在关键场景形成 AI 工作台。
