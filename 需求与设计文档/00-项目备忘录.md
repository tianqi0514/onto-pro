# 项目备忘录：Onto Pro

## 1. 项目背景

本项目探索一个“面向大模型推理的企业经营决策本体平台”。

最初的核心思考来自 `我的思考.pdf`：传统本体并不是完全无法表达复杂业务规则，OWL 2、SHACL、规则引擎和工业平台都能表达大量约束；真正的问题在于传统形式化本体主要面向确定性推理机、校验器和图查询，而不是面向大语言模型这个新的语义消费者。

因此，本项目的基本判断是：

> 本体负责定义，大模型负责推理，工具负责取数与计算，评测负责约束质量。

平台目标不是再造一个更强的规则引擎，而是构建一个企业级语义操作层，让 LLM 能在本体、知识库、结构化数据、skills、MCP 工具和工作流之间进行可追溯、可评测、可治理的经营决策分析。

## 2. 核心主张

### 2.1 本体的新职责

AI 时代的业务语义本体应收敛为定义层：

- 定义业务对象；
- 定义对象属性；
- 定义场景化关系；
- 定义指标口径；
- 定义行为和动作模板；
- 定义原子业务规则；
- 定义经营或审查场景；
- 定义可用数据源、skills 和证据要求。

本体不承担主要推理职责。真正的推理由大模型根据具体场景完成。

### 2.2 大模型的新职责

大模型负责：

- 识别用户问题属于哪个业务场景；
- 选择相关对象、关系、规则和指标；
- 规划需要调用哪些数据源或 skills；
- 组装原子规则形成推理链；
- 解释计算结果和业务影响；
- 输出结论、证据、推理轨迹和建议动作。

### 2.3 工具和 skills 的职责

大模型不应承担精确大规模计算。平台应把计算和取数交给工具：

- SQL 查询；
- 指标计算；
- 批量校验；
- 物料齐套检查；
- 产能测算；
- 排程模拟；
- 风控模型；
- 报告生成；
- 外部系统动作。

关键边界：大模型决定“该查什么、算什么、套用什么规则、如何解释”，工具负责可靠执行。

### 2.4 评测是一等公民

由于大模型推理没有传统逻辑系统的确定性保证，平台必须用评测替代形式证明：

- 场景匹配准确率；
- 规则选择准确率；
- 工具调用正确性；
- 结论准确性；
- 推理轨迹完整性；
- 证据引用正确性；
- 多次运行一致性；
- 本体变更后的回归表现。

没有评测集，就不能声称本体覆盖了完整业务语义。

## 3. 当前关键设计决策

### 3.1 场景层是平台核心

当前判断：平台的一等中心不是对象、图谱、Agent 或 RAG，而是 `Scenario`。

场景决定：

- 激活哪些对象；
- 激活哪些关系；
- 激活哪些规则；
- 激活哪些指标；
- 可调用哪些数据源和 skills；
- 输出需要什么结构；
- 如何构造评测集。

场景层用于降低上下文复杂度，避免大模型在完整企业本体中“乱抓一气”。

### 3.2 规则必须原子化

规则不写成复杂巨型规则，而是拆成可组合的原子规则：

- 一条规则只做一个最小判断；
- 尽量只跨一跳关系；
- 带场景标签；
- 带引用对象、关系、属性；
- 带至少一个解好的示例；
- 能被评测。

复合推理由大模型在场景中组装多条原子规则完成。

### 3.3 可追溯优先于可判定

平台接受 LLM 非确定性，但要求输出结构化推理轨迹：

- 匹配了哪个场景；
- 使用了哪些事实；
- 激活了哪些规则；
- 调用了哪些工具；
- 每一步得到了什么中间结论；
- 最终结论依赖哪些证据；
- 哪些地方存在不确定性或缺失数据。

## 5. MVP 方向

当前最小闭环建议：

1. 上传文档；
2. 接入结构化数据 schema；
3. 从文档和数据生成本体草案；
4. 人工审核对象、关系、规则、场景；
5. 发布本体版本；
6. 用户提问；
7. Agent 匹配场景；
8. Agent 调用数据源或 skills；
9. Agent 组装原子规则推理；
10. 输出结论、证据和推理轨迹；
11. 用户反馈；
12. 沉淀评测用例；
13. 本体变更后跑回归测试。

## 6. 样板场景摘要

### 6.1 运输合规审查

典型问题：

> 成品 A 包含 B，B 包含 C，C 是危险品，A 原产国为中国。A 出库运输前是否需要特殊运输条件？

核心对象：

- Material；
- Factory；
- ShipmentOrder。

核心规则：

- 危险性一跳传递；
- 产地叠加触发；
- 特殊运输条件判断。

价值：

- 边界清楚；
- 规则文档容易准备；
- 数据结构简单；
- 适合验证第一版闭环。

### 6.2 锂电行业生产计划排程

典型问题：

> 下周有 3 个高优先级电芯订单，当前 A 产线周三停机维护，正极材料库存不足 2 天，是否需要调整生产计划？如果需要，应该如何调整？

核心对象：

- ProductionOrder；
- ProductModel；
- Material；
- WorkCenter；
- ProcessStep；
- MaintenanceWindow。

核心关系：

- requiresMaterial；
- followsProcess；
- canRunOn；
- blockedByMaintenance。

核心 skills：

- query_production_orders；
- check_material_availability；
- query_work_center_capacity；
- simulate_schedule；
- identify_bottlenecks。

关键边界：

> 大模型不直接求解复杂排程优化，而是负责识别场景、组装约束、调用排程或计算 skill、解释结果和生成可追溯的调整建议。

### 6.3 金融合规与风控审查

适合子场景：

- 客户准入尽调审查；
- 交易异常初筛；
- 授信材料完整性审查；
- 贷后风险预警解释；
- 反洗钱可疑线索辅助分析；
- 投顾或销售适当性审查；
- 内控制度匹配与例外检查。

当前判断：

- 价值很高；
- 证据和审计要求强；
- 适合平台成熟后做专项样板；
- 第一版只能做辅助审查和风险提示，不能自动通过或拒绝业务。

## 7. 待验证问题

1. 场景层能否显著降低大模型推理复杂度？
2. 原子规则加组合推理在真实业务中能稳定到什么程度？
3. 本体生成和人工审核的成本是否低于传统知识工程？
4. 评测集能否有效约束大模型非确定性？
5. 外部工具计算与大模型推理的分工是否自然？
6. 场景数量增长后，场景匹配是否需要 GraphRAG 或类似检索机制？
7. 自然语言本体如何防止随业务演进而漂移？
8. 相比 GraphRAG、ontology-grounded extraction、神经符号企业知识图谱，本项目的实证增量在哪里？

## 8. 下一步建议

建议下一步产出：

1. 系统架构设计：前端、后端、数据存储、Agent runtime、MCP 层、评测系统。
2. 数据库 schema：围绕核心数据模型落 PostgreSQL 表结构。
3. 第一版样板数据：运输合规审查的物料、BOM、规则文档和评测用例。
4. Agent 推理协议：输入、场景匹配、工具计划、推理轨迹、输出 JSON schema。
5. 评测框架设计：如何打分、如何回归、如何比较版本。
